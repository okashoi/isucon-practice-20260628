# ISUCON 過去問 リファレンス

対象アプリのドメインに近い回を探し、その回の典型ボトルネック・有効戦略を着想のヒントにする。
実際の改善は必ず対象アプリの実装事実と計測結果を優先すること。

## チューニング技術課題カタログ（よく出る論点）

ISUCONは単なるDBチューニングではなく、Webアプリ全体の **性能・整合性・スケーリング・仕様理解** を問う競技。改善アイデア出しの際は以下のカタログを対象アプリと突き合わせる。

### DB / クエリ

- **N+1クエリの解消**: 典型中の典型。JOIN、IN句一括取得、アプリ内キャッシュ、データ構造の持ち替えで潰す。（例: ISUCON9予選の商品一覧でカテゴリ/出品者/購入者のN+1）
- **インデックス不足の解消**: `WHERE` / `ORDER BY` / `JOIN` / 範囲検索に適切なindexを貼る。やみくもに貼ると書き込み性能・実行計画に悪影響。（ISUCON11予選で頻出指摘）
- **不要な全件取得・LIMIT不足**: ページング、最新N件、ランキング、時系列グラフで必要件数だけ取る。（ISUCON11予選でLIMIT未設定が典型ボトルネック）
- **集計処理の高速化**: ランキング/売上/課金/スコア/統計/トレンド/グラフを毎回SQL集計しない。事前計算・差分更新・集計テーブル化・キャッシュ。（ISUCON12予選の課金計算、MySQL/SQLiteまたぎ集計）
- **大量書き込みと読み込みの両立**: IoT/ログ/コンディション/チャット/コメント/履歴。書き込みを受けつつ最新状態・集計・グラフを高速返却。（ISUCON11予選）

### アプリケーション / ロジック

- **検索クエリの最適化**: 条件/範囲/地理/カテゴリ/複合検索。OR条件・LIKE・ソート・ページング・件数取得が重くなりやすい。（ISUCON10予選の物件検索、Bot・なぞって検索）
- **キャッシュ戦略**: マスタ/設定値/ユーザー情報/ランキング/統計/外部API結果。難所は「何を・いつまで・どう無効化するか」。更新頻度の低いデータはアプリ内メモリ/Redis/nginx/ファイル化が候補。
- **外部API・外部サービス呼び出しの削減**: 決済/配送/認証/スパム判定。遅い外部APIを同期/トランザクション中/過剰回数で呼ぶアンチパターン。（ISUCON9予選: 配送APIにベンチ中0.8秒遅延）
- **トランザクション・ロック競合**: 購入/予約/座席確保/在庫/マッチング/ゲーム状態更新。高速化しつつ二重購入・二重予約・不正な状態遷移を防ぐ。（ISUCON9予選: 1商品への購入集中＋外部API含むTXのロック待ち）
- **認証・ログイン処理の負荷**: bcrypt/JWT検証/セッション管理/ログインAPI集中。セキュリティを壊さずキャッシュ・処理分離・専用サーバ化。（ISUCON9予選: bcryptが最後の負荷、ログイン専用サーバ案）
- **リアルタイム・準リアルタイム処理**: チャット/コメント/通知/ライブ配信/SSE/ポーリング/マッチング。「毎回DBを見に行く」実装が重い。（ISUCON13ライブ配信、ISUCON14ライドシェア）
- **状態遷移・ドメインロジックの理解**: 予約/購入/配送/配車/ゲーム進行/講義提出。仕様を読み違えるとスコアが出ない・失格・不整合になる。（ISUCON14はマッチング含むアプリ全体最適化）

### インフラ / 構成

- **複数台構成への移行**: app / DB / nginx / Redis / 静的配信をどう分けるか。1台目にだけベンチが来る構成では他サーバへ振り分ける。（ISUCON9予選: 最大3台、1台目から振り分け）
- **DB分割・データ配置の最適化**: appとDBの分離、テナント/ユーザー/日付単位の分割、SQLite→MySQL移行 or SQLiteのまま分散の判断。（ISUCON12予選: MySQL移行でアプリ2台+MySQL1台が伸ばしやすい）
- **静的ファイル・画像・アイコン配信の最適化**: アプリが画像/アセットを返してCPU/IOを食う。nginx直接配信、ファイル配置変更、キャッシュヘッダ、CDN相当へ寄せる。
- **HTTP/nginx/リバースプロキシのチューニング**: keep-alive / gzip / HTTP/2 / 静的配信 / upstream / worker設定。（ISUCON9予選: nginxほぼデフォルト、HTTP/2有効化等が必要）

### 進め方（メタ）

- **ベンチマーカーの行動理解**: どのAPIがどの順序で呼ばれるか、何がスコアに効くか、どのエラーが減点か。（ISUCON11予選: ISU送信/ユーザー確認・改善・グラフ確認のロール）
- **計測にもとづく優先順位付け**: access log / slow query log / pprof / top / iostat / pt-query-digest / alp でボトルネックを確認。典型パターンを直してもスコアが伸びるとは限らない。（ISUCON11予選: 計測の大切さを伝える設計）

### 攻略の基本サイクル

「SQLを速くする力」だけでは不足。出題は以下の組み合わせ:

- DBの基本性能問題 / アプリロジックの無駄 / キャッシュ・事前計算・非正規化 / 複数台構成・データ配置 / 外部API・ロック・整合性 / ベンチマーカー仕様の読み解き / 実サービスに近いドメイン理解

実戦サイクル: **計測 → ボトルネック特定 → 仕様確認 → 小さく修正 → ベンチ → ロールバック可能にする** を高速に回す。

## ドメイン別の着想ヒント

- 配車 / マッチング / 状態遷移（ISURIDE）→ ドメインロジックの書き換え、マッチング処理の最適化、SSE/ポーリング負荷の削減。
- ライブ配信 / 集計 / 多テナント（ISUPipe）→ 統計・集計の事前計算、サブドメインごとの処理、アイコン等の配信最適化。
- 放置ゲーム / 負荷集中API（ISU CONQUEST）→ MySQL 水平分割、ログイン処理の最適化、マスタデータのキャッシュ。
- マルチテナント（ISUPORTS）→ テナント分散、ロック（flock）解消、課金集計の事前確定、ID採番。
- 時系列集計 / トレンド（ISUCONDITION）→ 大テーブルの書き込み/読み込み分離、トレンドキャッシュ、limit調整、バルクインサート。
- 講義管理 / シナリオ依存（ISUCHOLAR）→ シナリオ全体の律速判定、エンドポイント間依存の解消。
- 検索 / 位置情報（ISUUMO）→ インデックス設計、検索クエリ最適化、DB分割。
- 座席 / 区間予約（ISUXPRESS, チケット予約）→ 条件判定を保ったクエリ削減、ロック競合・レースコンディション対策、日付パーティショニング。
- フリマ / 外部API（ISUCARI）→ 外部API連携の最適化、SPA向けAPIのN+1解消。
- チャット / 画像配信（ISUCON7予選）→ 画像配信の複数台分散、nginx try_files、Redis活用。
- マイクロサービス連携（ISUCON6予選）→ プロセス間通信の削減、サブアプリ統合、自動リンク処理の高速化。
- SNS / 大量N+1（ISUxi）→ N+1解消、JOIN化、全件キャッシュ、ボトルネックの優先順位付け。
