# ISUCON Tuning Idea

## 目的

対象アプリケーションの既存実装を調査し、ISUCON のスコア（ベンチマークのスループット）を上げるための具体的なチューニングアイデアを提案する。

提案は「現在の実装の事実」と「計測結果（あれば）」と「過去問で頻出のボトルネックパターン」を根拠にする。推測だけで断定しない。HTTP API の互換性は壊さない前提で提案する。

## スコープ

- やること: 既存実装の調査、ボトルネック仮説の提示、改善アイデアの優先順位付け、各案の効果見積もり・リスク・検証方法の提示。
- やらないこと: ユーザーの承認なしに大規模な実装変更を勝手に行うこと。HTTP API のリクエスト/レスポンス互換を壊す提案を無断で進めること。

## ワークフロー

1. 計測の有無を確認する。
   - alp / kataribe などのアクセスログ集計、MySQL slow query log（`mybin/mysql/mysql_slow_query.sh`）、`pprof`、`top`/`dstat` 等の結果があれば最優先で読む。
   - 計測結果が無い場合は「まず計測を取るべき」と明記し、計測手順を提案したうえで、コード読解ベースの仮説提案に切り替える。
2. アプリ構成を把握する。
   - 言語/フレームワーク、ルーティング、handler、middleware、DB スキーマ、初期データ量、外部API連携、静的配信、セッション/キャッシュの有無。
3. ボトルネック仮説を立てる。
   - 律速エンドポイント（呼ばれる回数 × 1回あたりのコスト）を特定する。
   - 下記「典型ボトルネックのチェックリスト」と `reference.md` の過去問パターンを突き合わせる。
4. 改善アイデアを優先順位付けする。
   - 各案について: 効果（大/中/小）、実装コスト、リスク（互換性・整合性）、検証方法をセットで示す。
   - 「効果が大きく、コスト・リスクが小さい」ものを上に並べる。
5. 提案を出力する（後述の出力フォーマット）。ユーザーの承認後に実装へ進む。

## 攻略の前提

ISUCON は単なる DB チューニングではなく、Webアプリ全体の **性能・整合性・スケーリング・仕様理解** を問う競技。「SQL を速くする力」だけでは不足する。常に次のサイクルを高速に回す前提でアイデアを出す:

**計測 → ボトルネック特定 → 仕様確認 → 小さく修正 → ベンチ → ロールバック可能にする**

よく出る技術課題の網羅カタログ（N+1、index、LIMIT、集計、大量書き込み、検索、キャッシュ、外部API、ロック競合、認証負荷、リアルタイム、状態遷移、複数台構成、DB分割、静的配信、nginx、ベンチマーカー理解、計測優先）は `reference.md` の「チューニング技術課題カタログ」に整理してある。アイデア出しの際は必ず突き合わせること。

## 典型ボトルネックのチェックリスト

DB / クエリ:
- N+1 クエリ（ループ内 SELECT、関連取得）→ JOIN / IN 一括取得 / プリロード。
- 不足インデックス、複合インデックス、カバリングインデックス、`ORDER BY` + `LIMIT` の最適化。
- `SELECT *` の削減、不要カラム取得の削減。
- 集計・ランキングの都度計算 → 事前集計・カウンタテーブル・更新時計算。
- トランザクション/ロック競合、`FOR UPDATE` 範囲、デッドロック。
- コネクション数（`SetMaxOpenConns` 等）、prepared statement、bulk insert。

アプリケーション:
- 同期的な外部API/重い処理 → 非同期化・バッチ化・キャッシュ。
- 同一リクエスト内・リクエスト間の重複計算 → メモ化・キャッシュ（プロセス内 / Redis）。
- パスワードハッシュ等 CPU 負荷の高い処理のコスト調整（仕様が許す範囲で）。
- JSON シリアライズ、不要なログ出力、デバッグ出力の削減。

ミドルウェア / インフラ:
- 静的ファイル配信を nginx へ寄せる、画像等のアプリ経由配信を避ける。
- nginx のキャッシュ、gzip、keepalive、`open_file_cache`。
- MySQL のバッファプール、`innodb_flush_log_at_trx_commit`、ログ設定。
- 複数台構成: DB 分離、アプリ複数台、セッション/キャッシュの共有（Redis）、ロードバランス。
- 画像・大きなデータの保存先（DB → ファイル/オブジェクトストレージ/ホスト分散）。

## 過去問パターンの参照

過去 ISUCON の題材ごとの「何が問われたか」「公式講評・解説記事リンク」は `reference.md` にまとめてある。

対象アプリのドメイン（配車/マッチング、ライブ配信/集計、放置ゲーム/水平分割、マルチテナント、時系列集計/トレンド、座席予約/区間判定、フリマ/外部API、チャット/画像配信 など）に近い回を `reference.md` から探し、その回で有効だった戦略を着想のヒントにする。ただし対象アプリの実装事実を必ず優先する。

## 出力フォーマット

1. 調査サマリ: 構成・データ量・（あれば）計測結果から見えた事実。
2. ボトルネック仮説: 律速箇所と根拠（ログ行/コード箇所/クエリ）。
3. 改善アイデア一覧（優先順位順）。各項目:
   - 概要 / 対象箇所
   - 期待効果（大・中・小）と理由
   - 実装コスト・リスク（互換性/整合性への影響）
   - 検証方法（ベンチ前後比較、計測指標）
4. 推奨する次の一手（最初に着手すべき1〜2件）。

計測が無い場合は冒頭で必ず「まず計測（alp / slow query / pprof）を取ること」を推奨する。
